BibleCMS Frontend Usage Guide
Last updated: 2026-06-19

This file is written for the frontend developer working on the dashboard UI.
It explains which backend endpoints exist, what they return, how they are used
in the dashboard, and what to watch out for when wiring new screens.

Base URL
--------
- `https://16-171-22-191.sslip.io`

Important notes
---------------
1. The current runtime routes in `index.js` are the source of truth.
2. `openapi.json` contains some older route names for auth:
   - documented there: `/api/auth/register`, `/api/auth/login`
   - actual code routes: `/api/register`, `/api/login`
3. Most admin list endpoints are public in the current code.
4. `User-Agent` filtering is not enforced in the current runtime, even though
   some docs still mention it.
5. File uploads use `multipart/form-data` and are handled with Multer.
6. Dashboard tables are reloaded with jQuery AJAX and DataTables.

Base app routes
---------------
- `GET /`
  - Redirects to `/login`
- `GET /login`
  - Renders the login page
- `GET /dashboard`
  - Renders the dashboard page
- `GET /register`
  - Redirects to `/login`

Global API behavior
-------------------
- JSON requests use `express.json()`
- Uploaded files are served from `/uploads`
- CORS is open
- Helmet is enabled, but CSP is relaxed for CDN assets
- Body strings are sanitized before controller logic runs

Auth endpoints
--------------
- `POST /api/register`
  - Used for user registration
  - Body fields:
    - `firstName`
    - `lastName`
    - `email`
    - `phoneNumber`
    - `password`
  - Response includes created user

- `POST /api/login`
  - Used for authentication
  - Body fields:
    - `email`
    - `password`
  - Response includes `token` and `user`

- `GET /api/auth/verify`
  - Verifies the current token via controller logic

Catalog API usage
------------------
List catalogs:
```http
GET /api/catalogs
```

Create catalog:
```http
POST /api/catalogs
Content-Type: multipart/form-data
```

Expected form fields:
- `name` string
- `description` string
- `verticalImage` file
- `horizontalImage` file

Dashboard behavior:
- `loadCatalogs()` fetches the list
- After create success, the UI reloads catalogs
- The create form relies on two file inputs

Payments API usage
------------------
- `GET /api/payments`
- `POST /api/payments`
- `POST /api/payments/stripe-checkout`
  - Example: `https://16-171-22-191.sslip.io/api/payments/stripe-checkout`

Expected fields for create:
- `uid` or `userId` or `uuid`
- `userEmail`
- `productId`
- `amount`
- `currency`
- `status`
- `transactionId`
- `uuid`
- `subscriptionEndDate`
- `ip`
- `location`

Important behavior:
- `uid` is required by the runtime logic
- If `subscriptionEndDate` exists, subscription checks use it first
- Dashboard list shows:
  - payment id
  - user id
  - user email
  - product id
  - amount
  - currency
  - status
  - transaction id
  - created date

Stripe checkout behavior:
- Send `provider: "stripe"` or call `/api/payments/stripe-checkout`
- The backend returns:
  - `checkoutUrl`
  - `checkoutSessionId`
- Frontend should redirect the user to `checkoutUrl`
- Typical flow:
  1. Collect payment intent data on the frontend
  2. POST it to the backend
  3. Receive `checkoutUrl`
  4. Redirect with `window.location.href = checkoutUrl`

Subscription API usage
----------------------
- `GET /api/subscription/check`
- `POST /api/subscription/check`

Accepted identifiers:
- `uid`
- `userId`
- `uuid`

Response behavior:
- Returns `active` if subscription is valid
- Returns `exhausted` if missing, expired, or inactive

Dashboard note:
- This is not currently a sidebar section, but it is useful for app-side entitlement checks

DataTables behavior
-------------------
The dashboard uses DataTables for list-heavy sections.

Implementation pattern:
- Destroy existing table if already initialized
- Replace `<tbody>` rows
- Recreate DataTable

Typical pattern used in code:
1. `if ($.fn.DataTable.isDataTable('#tableId')) { $('#tableId').DataTable().destroy(); }`
2. Fetch fresh data with AJAX
3. Build HTML rows
4. Set `$('#tableBody').html(rows)`
5. Reinitialize DataTable

Frontend integration advice
---------------------------
1. Keep route names aligned with `index.js`, not with stale docs.
2. If you add a new section, add:
   - sidebar item
   - `showSection(...)` handler
   - loader function
   - table body target
   - create/edit/delete handlers as needed
3. Prefer `FormData` for any endpoint that accepts files.
4. After create/update/delete, reload the active section to keep the UI in sync.
5. Escape any values inserted into inline `onclick` handlers.
6. If a route uses `multipart/form-data`, do not set `contentType` manually in jQuery.
7. Use `Authorization: Bearer <token>` only for `/api/aws/upload` in the current codebase.
